# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Personal bilingual (ru/en) blog for Anton Vorontsov, built as a Hugo static site. Deployed to GitHub Pages at `vorontsov.dev`. No JS framework, no CSS framework, no npm/node, no Hugo Pipes тАФ plain HTML/CSS/JS and Go templates only. Keep it that way; do not introduce a build pipeline, bundler, or third-party CSS/JS library without being asked.

`legacy-site/` is the pre-Hugo version of the site (hand-written HTML/CSS/JS), kept only for reference/rollback. It is not built, not deployed, and nothing in `layouts/`/`static/` should link to it.

## Commands

- `hugo server` тАФ dev preview at http://localhost:1313 (also runnable via the `.claude/launch.json` config named `hugo-server`, e.g. through the Preview tool).
- `hugo` тАФ one-off build, outputs to `public/` (gitignored). No `baseURL` is configured locally (see "Domain handling" below), so links stay relative and `public/*.html` can be opened directly via `file://`.
- Adding a new post: see `INSTRUCTIONS.md` тАФ it covers both the normal bilingual case and RU-only/EN-only posts.
- No lint/test commands тАФ this is a content site, not an application.
- CI builds with `hugo --gc --minify --baseURL "<pages-url>/"` (see `.github/workflows/hugo.yml`); it installs Hugo Extended via a pinned version (`HUGO_VERSION` env var), not `hugo-bin`/npm.

## Domain handling тАФ the only place `vorontsov.dev` may appear is `static/CNAME`

`hugo.toml` has no `baseURL`. This is intentional: the production domain is owned entirely by GitHub Pages/Actions, not by the repo's Hugo config. Two consequences:
- Local builds (`hugo`, `hugo server`) fall back to a harmless `http://localhost/`-style baseURL, so a locally-built `public/index.html` opened via `file://` never redirects out to the live production site тАФ it was doing exactly that before this was fixed (an absolute `https://vorontsov.dev/...` baked into the generated redirect/canonical links).
- CI supplies the real URL at build time via `actions/configure-pages` + `--baseURL` (see `.github/workflows/hugo.yml`) тАФ never hardcoded in this repo.
- `static/CNAME` (containing just `vorontsov.dev`) is the **one** exception тАФ GitHub Pages custom domains require it, and this is the standard, expected place for it to live. Don't add the domain anywhere else (don't reintroduce `baseURL` in `hugo.toml`, don't hardcode it in a template or a hand-written static redirect).

## Bilingual content model (read this before touching any content or template)

The site uses Hugo's built-in multilingual mode. **Russian is the default, unprefixed language** (`/`, `/about.html`, `/posts/<slug>.html`); **English is prefixed** (`/en/`, `/en/about.html`, `/en/posts/<slug>.html`). This was a deliberate choice, made after initially trying both languages prefixed (`/ru/...`/`/en/...`) тАФ that turned out to be worse UX for the primary (Russian) audience, and switching the default language to unprefixed also happens to sidestep a real Hugo defect (see "URL scheme and rendering quirks" below). The language switcher navigates to a real translated page rather than toggling visibility client-side тАФ there is no client-side language state at all anymore. This replaced an earlier custom `|||`-marker scheme; see git history before the `feature/hugo-native-i18n-migration` branch for that.

- **Content organization**: one file per language per page, sharing a base filename тАФ `content/posts/<slug>.ru.md` / `content/posts/<slug>.en.md`. Hugo pairs translations automatically by matching base filename; no `translationKey` needed. A post can have just one language file тАФ see `INSTRUCTIONS.md` for what that does (the page simply doesn't exist on the other language's site; the switcher falls back to that language's home page instead of a broken link).
- **Languages are configured in `hugo.toml`**: `defaultContentLanguage = "ru"`, `defaultContentLanguageInSubdir = false`, and `[languages.ru]` / `[languages.en]` blocks (each with its own `title`, `label`, `locale`, `weight`, and `params.profileName`/`params.bio`). Params shared across both languages (e.g. `profileTitle`) live in the top-level `[params]`.
- **Nav menu** is split into `config/_default/menus.ru.toml` / `menus.en.toml` (`[[main]]` entries with `name`, `pageRef`, `weight`). `layouts/partials/header.html` renders `.Site.Menus.main`, resolving each entry's active/highlighted state by comparing `.Page.RelPermalink` (and, for the Posts entry, by section).
- **UI strings** (nav aria-labels, theme toggle text, section headings, footer copyright, etc.) live in `i18n/ru.toml` / `i18n/en.toml`, called via `{{ i18n "key" }}` (optionally with a `(dict ...)` argument, e.g. `footerCopyright` takes `Year`/`Name`). Any new hardcoded UI string in a template should go through i18n instead тАФ check both TOML files get a matching key.
- **The language switcher is a real link**, not a JS toggle: `layouts/partials/header.html` uses `.IsTranslated` / `.Translations` to link to the current page's actual translated `.RelPermalink` (falling back to the other language's home page only if the current page genuinely has no translation, via `.Sites`). `static/js/modules/navigation.js` only handles the theme toggle and TOC scrollspy now.
- **Theme toggle text** is still client-driven (dark/light is a per-visitor, not per-language, preference) тАФ the two possible labels are rendered server-side into `data-dark-label`/`data-light-label` attributes on `#theme-btn` (from the `themeDark`/`themeLight` i18n keys), and `navigation.js` just swaps between those two pre-localized strings. It never needs to know the current language.
- Post headings are plain single-language Markdown (`## Context {#context}`), no `|||` marker. `layouts/_default/_markup/render-heading.html` only keeps a duplicate-anchor guard (`errorf` on a repeated `{#slug}` within one page) тАФ the explicit ASCII `{#slug}` is still required: Goldmark's auto-generated id from Cyrillic text breaks the JS scrollspy's exact-id lookup in `navigation.js`.
- The TOC sidebar (`layouts/posts/single.html`) is built from `.Fragments.Headings` rather than `.TableOfContents` (`.Fragments` gives the h2тЖТh3 nesting the sidebar needs).
- Front matter is single-language per file тАФ no `foo`/`foo_en` pairing. `title`, `description`, `og_description` are plain fields on each language's own file. See `INSTRUCTIONS.md` for the full front-matter reference for new posts.
- **`description`/`og_description` on the three "root" pages (`_index.md`, `about.md`, `posts/_index.md`) never had an English counterpart even in the old marker scheme** тАФ those meta tags were always Russian-only regardless of selected language. `content/_index.en.md`, `content/about.en.md`, and `content/posts/_index.en.md` currently reuse the Russian `description`/`og_description` text verbatim rather than inventing a translation. Writing real English copy for these three is an open content task, not a template/build issue.

## URL scheme and rendering quirks

- `uglyURLs = true` reproduces the site's flat URL scheme (`/about.html`, `/posts/<slug>.html`, no `/about/` directories) for **both** languages. Combined with `defaultContentLanguage = "ru"` + `defaultContentLanguageInSubdir = false`, Russian content lands unprefixed at the site root and English gets the `/en/` prefix automatically тАФ no per-page `url:` override needed for `about.*.md` or individual posts (Hugo derives the right flat filename from `uglyURLs` + the language's prefix rule on its own).
- The one page that still needs an explicit `url:` override is the posts list: `content/posts/_index.ru.md` sets `url: /posts.html`, `content/posts/_index.en.md` sets `url: /en/posts.html` тАФ both also keep `outputs: [html]`, otherwise Hugo lets the section's RSS feed silently claim that URL instead of the HTML page. If you add another section-style archive/list page, it'll need the same two front-matter keys.
- **Do not set `defaultContentLanguageInSubdir = true`** (i.e. don't try to prefix Russian as `/ru/...`) without re-reading this section first. That configuration was tried during the original migration and hit a real, confirmed Hugo defect: with `uglyURLs = true`, Hugo's automatic root (`/`) тЖТ default-language-home redirect and the real default-language home page both resolve to the identical output path, and the auto-generated redirect stub silently clobbers the real rendered home page. There's no config flag to disable it (a PR proposing one тАФ `gohugoio/hugo` #6138 тАФ was never merged). Keeping the default language unprefixed avoids the whole mechanism, since there's no cross-language root redirect to generate in the first place.
- `relativeURLs = true` is load-bearing: it makes every internal link and asset path (`relURL`, `.RelPermalink`) dot-relative (`../css/styles.css`) instead of root-relative (`/css/styles.css`), so the generated `public/` folder can be opened directly via `file://` with no server, in addition to working when actually deployed. Don't remove this.
- `.Permalink` (used for `og:url`, hreflang) stays absolute-*path* regardless of `relativeURLs` тАФ with no `baseURL` configured locally, that means things like `og:url` render as root-relative (`/about.html`) rather than a full URL in local builds. That's expected and harmless (meta tags aren't navigated), and CI's `--baseURL` flag makes them fully qualified in the real deployed output.
- `[markup.goldmark.parser.attribute] title = true` in `hugo.toml` is what enables the `{#slug}` syntax on headings; `[markup.goldmark.renderer] unsafe = true` is what allows raw HTML in post intro paragraphs (e.g. `<strong>` inside about-page descriptions).
- `head.html` renders `hreflang` alternate links via `{{ range .AllTranslations }}` (includes self-reference, which is correct SEO practice).

## Template structure

- `layouts/_default/baseof.html` computes `$isPostSingle` (`and .IsPage (eq .Section "posts")`) once via `.Scratch` and reuses it (via `.Scratch.Get`) in `layouts/partials/head.html` тАФ a post-single page gets a title suffix, `container-wide`, and no OG meta tags; other page kinds don't. If you add another page kind that needs different `<head>`/container behavior, extend this pattern rather than adding another ad hoc boolean. `<html lang="...">` comes from `.Language.Lang`.
- `layouts/partials/profile-header.html` (shared by `layouts/index.html` and `layouts/about.html`) derives whether to show the avatar/social-links block from `.Params.social` being present тАФ not from an explicit flag passed in. If a future page needs the avatar without social links (or vice versa), this coupling will need revisiting.
- `content/about.ru.md` / `about.en.md` are structured data, not prose: `experience`, `speaking` (translated per-language тАФ `role`/`date`/`desc` fields, plus a shared untranslated `company` on `experience` entries), `skills`/`social` (untranslated тАФ English tech terms and proper nouns, duplicated identically in both language files) are YAML lists in front matter, rendered by `layouts/about.html` (routed via `layout: "about"`) using `range`. `layouts/partials/timeline-item.html` renders one `experience`/`speaking` entry, parameterized by `showCompany`.
- Russian date formatting (genitive month names, e.g. "1 ╨╕╤О╨╜╤П 2026") has no Go/Hugo built-in тАФ `layouts/partials/date-ru.html` hardcodes the month list, called only when `.Language.Lang` is `"ru"`. English dates use plain `.Date.Format`.
- Taxonomy `tags` is declared in `hugo.toml` for future use; there's no `layouts/taxonomy/`/`layouts/term/` template yet and posts just carry `tags: []`. `disableKinds = ["taxonomy", "term"]` (also in `hugo.toml`) stops Hugo from generating `/tags/` pages for it in the meantime тАФ remove that line if/when real tag-browsing UI gets built. Don't wire up tag pages unless asked.

## Deployment

- `.github/workflows/hugo.yml` deploys on push to `master` (the working branch is `blog` тАФ merge before expecting a deploy) or manual `workflow_dispatch`, via the official `actions/upload-pages-artifact` + `actions/deploy-pages` flow (no `gh-pages` branch).
- Deploys are expected to be infrequent тАФ the Hugo install step is intentionally a plain `wget`+`dpkg`, not cached; don't add caching back without a reason.
- Custom domain is `vorontsov.dev` via `static/CNAME` тАФ the only place that domain is allowed to appear in this repo (see "Domain handling" above). `baseURL` is never set in `hugo.toml`; CI overrides it per-build to whatever `actions/configure-pages` reports for the live Pages URL.
