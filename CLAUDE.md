# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Personal bilingual (ru/en) blog for Anton Vorontsov, built as a Hugo static site. Deployed to GitHub Pages at `vorontsov.dev`. No JS framework, no CSS framework, no npm/node, no Hugo Pipes — plain HTML/CSS/JS and Go templates only. Keep it that way; do not introduce a build pipeline, bundler, or third-party CSS/JS library without being asked.

`legacy-site/` is the pre-Hugo version of the site (hand-written HTML/CSS/JS), kept only for reference/rollback. It is not built, not deployed, and nothing in `layouts/`/`static/` should link to it.

## Commands

- `hugo server` — dev preview at http://localhost:1313 (also runnable via the `.claude/launch.json` config named `hugo-server`, e.g. through the Preview tool).
- `hugo` — one-off build, outputs to `public/` (gitignored). No `baseURL` is configured locally (see "Domain handling" below), so links stay relative and `public/*.html` can be opened directly via `file://`.
- Adding a new post: see `INSTRUCTIONS.md` — it covers both the normal bilingual case and RU-only/EN-only posts.
- Validating a post before publishing: delegate to the `blog-post-validator` subagent (via the Agent tool) — it has scoped access to the `validate-blog-post` skill (`.claude/skills/validate-blog-post/SKILL.md`). Don't invoke that skill directly from the main conversation.
- Adding an existing post to a series: delegate to the `series-editor` subagent (via the Agent tool) — it has scoped access to the `add-post-to-series` skill (`.claude/skills/add-post-to-series/SKILL.md`). Don't invoke that skill directly from the main conversation.
- No lint/test commands — this is a content site, not an application.
- CI builds with `hugo --gc --minify --baseURL "<pages-url>/"` (see `.github/workflows/hugo.yml`); it installs Hugo Extended via a pinned version (`HUGO_VERSION` env var), not `hugo-bin`/npm.

## Domain handling — the only place `vorontsov.dev` may appear is `static/CNAME`

`hugo.toml` has no `baseURL`. This is intentional: the production domain is owned entirely by GitHub Pages/Actions, not by the repo's Hugo config. Two consequences:
- Local builds (`hugo`, `hugo server`) fall back to a harmless `http://localhost/`-style baseURL, so a locally-built `public/index.html` opened via `file://` never redirects out to the live production site — it was doing exactly that before this was fixed (an absolute `https://vorontsov.dev/...` baked into the generated redirect/canonical links).
- CI supplies the real URL at build time via `actions/configure-pages` + `--baseURL` (see `.github/workflows/hugo.yml`) — never hardcoded in this repo.
- `static/CNAME` (containing just `vorontsov.dev`) is the **one** exception — GitHub Pages custom domains require it, and this is the standard, expected place for it to live. Don't add the domain anywhere else (don't reintroduce `baseURL` in `hugo.toml`, don't hardcode it in a template or a hand-written static redirect).

## Bilingual content model (read this before touching any content or template)

The site uses Hugo's built-in multilingual mode. **Russian is the default, unprefixed language** (`/`, `/about.html`, `/posts/<slug>.html`); **English is prefixed** (`/en/`, `/en/about.html`, `/en/posts/<slug>.html`). This was a deliberate choice, made after initially trying both languages prefixed (`/ru/...`/`/en/...`) — that turned out to be worse UX for the primary (Russian) audience, and switching the default language to unprefixed also happens to sidestep a real Hugo defect (see "URL scheme and rendering quirks" below). The language switcher navigates to a real translated page rather than toggling visibility client-side — there is no client-side language state at all anymore. This replaced an earlier custom `|||`-marker scheme; see git history before the `feature/hugo-native-i18n-migration` branch for that.

- **Content organization**: one file per language per page, sharing a base filename — `content/posts/<slug>.ru.md` / `content/posts/<slug>.en.md`. Hugo pairs translations automatically by matching base filename; no `translationKey` needed. A post can have just one language file — see `INSTRUCTIONS.md` for what that does (the page simply doesn't exist on the other language's site; the switcher falls back to that language's home page instead of a broken link).
- **Languages are configured in `hugo.toml`**: `defaultContentLanguage = "ru"`, `defaultContentLanguageInSubdir = false`, and `[languages.ru]` / `[languages.en]` blocks (each with its own `title`, `label`, `locale`, `weight`, and `params.profileName`/`params.bio`). Params shared across both languages (e.g. `profileTitle`) live in the top-level `[params]`.
- **Nav menu** is split into `config/_default/menus.ru.toml` / `menus.en.toml` (`[[main]]` entries with `name`, `pageRef`, `weight`). `layouts/partials/header.html` renders `.Site.Menus.main`, resolving each entry's active/highlighted state by comparing `.Page.RelPermalink` (and, for the Posts entry, by section).
- **UI strings** (nav aria-labels, theme toggle text, section headings, footer copyright, etc.) live in `i18n/ru.toml` / `i18n/en.toml`, called via `{{ i18n "key" }}` (optionally with a `(dict ...)` argument, e.g. `footerCopyright` takes `Year`/`Name`). Any new hardcoded UI string in a template should go through i18n instead — check both TOML files get a matching key.
- **The language switcher is a real link**, not a JS toggle: `layouts/partials/header.html` uses `.IsTranslated` / `.Translations` to link to the current page's actual translated `.RelPermalink` (falling back to the other language's home page only if the current page genuinely has no translation, via `.Sites`). `static/js/modules/navigation.js` only handles the theme toggle and TOC scrollspy now.
- **Theme toggle text** is still client-driven (dark/light is a per-visitor, not per-language, preference) — the two possible labels are rendered server-side into `data-dark-label`/`data-light-label` attributes on `#theme-btn` (from the `themeDark`/`themeLight` i18n keys), and `navigation.js` just swaps between those two pre-localized strings. It never needs to know the current language.
- Post headings are plain single-language Markdown (`## Context {#context}`), no `|||` marker. The explicit ASCII `{#slug}` is required — Goldmark's auto-generated id from Cyrillic text breaks the JS scrollspy's exact-id lookup in `navigation.js`. `layouts/_default/_markup/render-heading.html` enforces the anchor **format** (`errorf` unless it matches `^[a-z0-9]+(-[a-z0-9]+)*$`); `layouts/posts/single.html` separately enforces **uniqueness** within a page while building the TOC from `.Fragments.Headings` (moved out of the render hook because `Page.Scratch` state leaked across Hugo dev-server render passes).
- The TOC sidebar (`layouts/posts/single.html`) is built from `.Fragments.Headings` rather than `.TableOfContents` (`.Fragments` gives the h2→h3 nesting the sidebar needs).
- Front matter is single-language per file — no `foo`/`foo_en` pairing. A regular post's schema: `title` (string), `date` (`YYYY-MM-DD`, same value on both language files), `description` (string, this language only), `tags` (`[]`, English slugs, untranslated — see the "Taxonomy tags" bullet under "Template structure" below), `series` (optional object `{name, number}`, untranslated `name` — see the "Series" bullet under "Template structure" below). `og_description` only appears on the three root pages (`_index.md`, `about.md`, `posts/_index.md`), never on a regular post. See `INSTRUCTIONS.md` for the full walkthrough of adding a new post.
- **`description`/`og_description` on the three "root" pages (`_index.md`, `about.md`, `posts/_index.md`) never had an English counterpart even in the old marker scheme** — those meta tags were always Russian-only regardless of selected language. `content/_index.en.md`, `content/about.en.md`, and `content/posts/_index.en.md` currently reuse the Russian `description`/`og_description` text verbatim rather than inventing a translation. Writing real English copy for these three is an open content task, not a template/build issue.

## URL scheme and rendering quirks

- `uglyURLs = true` reproduces the site's flat URL scheme (`/about.html`, `/posts/<slug>.html`, no `/about/` directories) for **both** languages. Combined with `defaultContentLanguage = "ru"` + `defaultContentLanguageInSubdir = false`, Russian content lands unprefixed at the site root and English gets the `/en/` prefix automatically — no per-page `url:` override needed for `about.*.md` or individual posts (Hugo derives the right flat filename from `uglyURLs` + the language's prefix rule on its own).
- The pages that still need an explicit `url:` override are the posts list (`content/posts/_index.ru.md` sets `url: /posts.html`, `content/posts/_index.en.md` sets `url: /en/posts.html`) and every series page under `content/series/` (see the "Series" bullet under "Template structure" below) — both also keep `outputs: [html]`, otherwise Hugo lets the section's RSS feed silently claim that URL instead of the HTML page. If you add another section-style archive/list page, it'll need the same two front-matter keys.
- **Do not set `defaultContentLanguageInSubdir = true`** (i.e. don't try to prefix Russian as `/ru/...`) without re-reading this section first. That configuration was tried during the original migration and hit a real, confirmed Hugo defect: with `uglyURLs = true`, Hugo's automatic root (`/`) → default-language-home redirect and the real default-language home page both resolve to the identical output path, and the auto-generated redirect stub silently clobbers the real rendered home page. There's no config flag to disable it (a PR proposing one — `gohugoio/hugo` #6138 — was never merged). Keeping the default language unprefixed avoids the whole mechanism, since there's no cross-language root redirect to generate in the first place.
- `relativeURLs = true` is load-bearing: it makes every internal link and asset path (`relURL`, `.RelPermalink`) dot-relative (`../css/styles.css`) instead of root-relative (`/css/styles.css`), so the generated `public/` folder can be opened directly via `file://` with no server, in addition to working when actually deployed. Don't remove this.
- `.Permalink` (used for `og:url`, hreflang) stays absolute-*path* regardless of `relativeURLs` — with no `baseURL` configured locally, that means things like `og:url` render as root-relative (`/about.html`) rather than a full URL in local builds. That's expected and harmless (meta tags aren't navigated), and CI's `--baseURL` flag makes them fully qualified in the real deployed output.
- `[markup.goldmark.parser.attribute] title = true` in `hugo.toml` is what enables the `{#slug}` syntax on headings; `[markup.goldmark.renderer] unsafe = true` is what allows raw HTML in post intro paragraphs (e.g. `<strong>` inside about-page descriptions).
- `head.html` renders `hreflang` alternate links via `{{ range .AllTranslations }}` (includes self-reference, which is correct SEO practice).

## Template structure

- `layouts/_default/baseof.html` computes `$isPostSingle` (`and .IsPage (eq .Section "posts")`) once via `.Scratch` and reuses it (via `.Scratch.Get`) in `layouts/partials/head.html` — a post-single page gets a title suffix, `container-wide`, and no OG meta tags; other page kinds don't. If you add another page kind that needs different `<head>`/container behavior, extend this pattern rather than adding another ad hoc boolean. `<html lang="...">` comes from `.Language.Lang`.
- `layouts/partials/profile-header.html` (shared by `layouts/index.html` and `layouts/about.html`) derives whether to show the avatar/social-links block from `.Params.social` being present — not from an explicit flag passed in. If a future page needs the avatar without social links (or vice versa), this coupling will need revisiting.
- `content/about.ru.md` / `about.en.md` are structured data, not prose: `experience`, `speaking` (translated per-language — `role`/`date`/`desc` fields, plus a shared untranslated `company` on `experience` entries), `skills`/`social` (untranslated — English tech terms and proper nouns, duplicated identically in both language files) are YAML lists in front matter, rendered by `layouts/about.html` (routed via `layout: "about"`) using `range`. `layouts/partials/timeline-item.html` renders one `experience`/`speaking` entry, parameterized by `showCompany`.
- Russian date formatting (genitive month names, e.g. "1 июня 2026") has no Go/Hugo built-in — `layouts/partials/date-ru.html` hardcodes the month list, called only when `.Language.Lang` is `"ru"`. English dates use plain `.Date.Format`.
- Taxonomy `tags` (`[taxonomies] tag = "tags"` in `hugo.toml`) powers per-tag post-listing pages: each post's `tags: []` front matter (untranslated — same English tag slugs in both language files) generates a `kind = "term"` page at `layouts/tags/term.html`, URL `/tags/<tag>.html` (ru, default/unprefixed) and `/en/tags/<tag>.html` (en — the site's usual language-prefix-at-front rule applies here too, since Hugo has no way to reposition it for auto-generated taxonomy pages). `[outputs] term = ["html"]` keeps it to just the HTML page — without it Hugo also emits an unused RSS feed per tag. It's a flat, non-year-grouped list (unlike `layouts/posts/list.html`) of title+date only (no description, no per-post tags), filtered to the term page's own language automatically by Hugo's per-language taxonomies; both it and `layouts/posts/list.html` render each row via the shared `layouts/partials/archive-item.html` partial. Term pages have no front matter to source `<title>`/meta description from, so `layouts/partials/head.html` special-cases `.Kind "term"` to use `#<tag>` and the `tagPageDescription` i18n string instead of `.Title`/`.Params.description` (which would otherwise be Hugo's auto title-cased term and an empty string). `layouts/partials/header.html` also special-cases `.Kind "term"` so the "Posts" nav item highlights on tag pages the same way it does on individual posts. The site-wide "all tags" overview page (`kind = "taxonomy"`, e.g. `/tags.html`) stays disabled via `disableKinds = ["taxonomy"]` — there's deliberately no such page. Tag links live only in `layouts/posts/single.html`'s `.post-meta` (via `.GetTerms "tags"`) and use the shared `.tag-pill` CSS class (also used by `.skill-tag` on the about page); the homepage's latest-posts cards and the archive list don't show tags — both intentionally out of scope (the homepage card is a single `<a class="card">` wrapping the whole block, so nested tag links would be invalid HTML).
- **Series** are a different mechanism from tags: not a taxonomy (auto-generated term pages), but a regular Hugo content section, `content/series/`, because each series is a hand-written page with its own title and optional description — like `about.md`, not like an auto-generated tag term. A post opts in via an optional `series: {name, number}` front-matter object (`name` an English hyphenated id matching a series page's filename, `number` this post's 1-based position in the series, unique within that series — enforced by a build-time `errorf` in `layouts/series/single.html`, the same pattern used for duplicate heading anchors). Series pages live at `content/series/<name>.ru.md` / `content/series/<name>.en.md` — **the usual filename-suffix bilingual convention, not a per-language subdirectory** (`content/series/en/<name>.md` would not be recognized as a translation at all, since this site has no per-language `contentDir` configured — translations are paired purely by matching base filename). The series id used to match posts is the file's base name (`.File.ContentBaseName`), so no separate id field is needed in the series page's own front matter. Each series page sets an explicit `url:` (`/series/<name>.html` ru, `/en/series/<name>.html` en — prefix-at-front, consistent with every other URL on the site) plus `outputs: [html]`, for the same RSS-suppression reason as `content/posts/_index.*.md`. It also sets a front-matter `description:` mirroring its body paragraph (the same pattern regular posts use) — `layouts/partials/head.html`'s generic `.Params.description` handling already covers the meta tag, and `layouts/index.searchindex.json` reads the same field, so without it a series' search result would show a title with no context. There is deliberately no site-wide "all series" page: `content/series/_index.ru.md` / `_index.en.md` set `build: {list: false, render: false}` to suppress Hugo's default auto-generated section list page, mirroring the absence of an "all tags" page. `layouts/series/single.html` renders the series' `.Title`/`.Content` (optional — may be empty) followed by its posts (`where .Site.RegularPages "Section" "posts"` filtered by `Params.series.name`, sorted by `Params.series.number`, per-language automatically since `.Site.RegularPages` is scoped to the current-language site); each row is one `.archive-link` reusing the `archive-item`/`archive-link` CSS classes, with the post's series number folded into the link text itself via the `seriesItemNumber` i18n string (`№1. <title>` ru / `#1. <title>` en) rather than a separate `.archive-date`-style element. A post that belongs to a series renders three separate places in `layouts/posts/single.html`, all resolving the series page via `.Site.GetPage (printf "series/%s" $seriesID)` with the same fallback if it returns nil (no such series page yet): link text falls back to the raw series id, link href falls back to the post's own `.RelPermalink` (self-link, never a broken one) — (1) `.post-series-link` under `.post-meta` in the post header, rendering the `postSeriesMeta` i18n string (`Серия постов «<name>» №<n>` ru / `Post series "<name>" #<n>` en), (2) a `.toc-series` block inside `.toc-sidebar` showing only the series title/link (no number — the number is already in (1)), hidden together with the rest of the sidebar on mobile via the existing `.toc-sidebar { display: none }` rule, (3) a separate `.post-series-mobile` link after `.post-content`, centered with symmetric top/bottom padding, shown only below the `767px` breakpoint (the mirror image of (2), since (2) only exists on desktop) using the `seriesMobileText` i18n string as one fully-clickable sentence (avoids embedding an `<a>` mid-translation, where ru/en word order differs). Series pages are included in the search index (`layouts/index.searchindex.json`) via the same `where .Site.RegularPages "Section" "..."` pattern already used for posts.

## Deployment

- `.github/workflows/hugo.yml` deploys on push to `master` (the working branch is `blog` — merge before expecting a deploy) or manual `workflow_dispatch`, via the official `actions/upload-pages-artifact` + `actions/deploy-pages` flow (no `gh-pages` branch).
- Deploys are expected to be infrequent — the Hugo install step is intentionally a plain `wget`+`dpkg`, not cached; don't add caching back without a reason.
- Custom domain is `vorontsov.dev` via `static/CNAME` — the only place that domain is allowed to appear in this repo (see "Domain handling" above). `baseURL` is never set in `hugo.toml`; CI overrides it per-build to whatever `actions/configure-pages` reports for the live Pages URL.
