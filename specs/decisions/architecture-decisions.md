# Архитектурные решения (ADR)

Значимые решения о дизайне системы: **что** решено, **почему**, и **какие
последствия**. Формат — облегчённый ADR. Все решения отражают текущее состояние
проекта.

---

## ADR-1. Русский — язык по умолчанию, без префикса {#adr-lang-default}

**Решение.** `defaultContentLanguage = "ru"`, `defaultContentLanguageInSubdir =
false`. Русский контент — в корне без префикса (`/`, `/about.html`), английский — с
префиксом `/en/`.

**Почему.**
1. **UX основной аудитории.** Изначально пробовали оба языка с префиксом (`/ru/…` /
   `/en/…`) — для основной, русскоязычной аудитории это хуже (лишний сегмент, редирект
   с корня).
2. **Обход дефекта Hugo.** С `defaultContentLanguageInSubdir = true` + `uglyURLs =
   true` авто-редирект `/` → домашняя страница языка по умолчанию и сама домашняя
   страница резолвятся в один выходной путь (`ru/index.html`), и авто-редирект молча
   затирает настоящую страницу. Флага отключить это нет (PR `gohugoio/hugo` #6138 не
   смёржен). «Дефолтный язык без префикса» убирает сам механизм — кросс-языкового
   редиректа с корня просто не возникает.

**Последствия.**
- Английские страницы всегда с `/en/`; ручной `url:` для en прописывается с префиксом.
- **Нельзя** переключать `defaultContentLanguageInSubdir` на `true` без перечтения
  этого ADR.

**См.** [`data-model/url-scheme.md`](../data-model/url-scheme.md),
[`conventions/bilingual-model.md`](../conventions/bilingual-model.md).

## ADR-2. Один файл на язык вместо маркер-схемы {#adr-file-per-lang}

**Решение.** Каждая страница — отдельный файл на язык (`foo.ru.md` / `foo.en.md`),
парятся по базовому имени. Встроенный многоязычный режим Hugo.

**Почему.** Раньше была кастомная схема с маркерами `|||` внутри одного файла
(двуязычные спаны, client-side переключение видимости). Это давало: сложный парсинг,
клиентское языковое состояние, невозможность иметь реально разные структуры на языках.
Нативный режим Hugo проще, даёт настоящие переведённые страницы и SEO-hreflang.

**Последствия.**
- Переключатель языка — реальная ссылка, клиентского языкового состояния нет.
- Пост может существовать на одном языке (просто нет второго файла).
- Историю старой схемы искать в git до ветки `feature/hugo-native-i18n-migration`.

**См.** [`conventions/bilingual-model.md`](../conventions/bilingual-model.md).

## ADR-3. Плоские `.html` URL + относительные ссылки {#adr-flat-urls}

**Решение.** `uglyURLs = true` (плоские `/about.html`, не `/about/`) +
`relativeURLs = true` (dot-relative ссылки).

**Почему.** Чтобы собранную папку `public/` можно было открыть напрямую через
`file://` без сервера — быстрая проверка билда, отсутствие зависимости от веб-сервера
для просмотра. Плоские `.html` + относительные пути делают это возможным.

**Последствия.**
- Разводящим страницам разделов (архив постов, серии) нужны явные `url:` +
  `outputs: [html]` — иначе RSS-фид уводит URL.
- `.Permalink` (для `og:url`/hreflang) остаётся абсолютным путём; локально без
  `baseURL` он корне-относительный — безвредно.

**См.** [`data-model/url-scheme.md`](../data-model/url-scheme.md).

## ADR-4. Домен только в `static/CNAME`, `baseURL` не задан {#adr-domain}

**Решение.** `hugo.toml` **без** `baseURL`. Домен `vorontsov.dev` — только в
`static/CNAME`. Реальный `baseURL` подставляет CI флагом `--baseURL`.

**Почему.** Продакшн-домен принадлежит GitHub Pages/Actions, а не конфигу репозитория.
С зашитым `baseURL` локально собранный `public/index.html`, открытый через `file://`,
редиректил на живой прод (в redirect/canonical был абсолютный `https://vorontsov.dev/…`).
Без `baseURL` локальные сборки откатываются на безобидный localhost.

**Последствия.**
- **Не** возвращать `baseURL` в `hugo.toml`, **не** хардкодить домен в шаблонах/редиректах.
- CI: `actions/configure-pages` → `--baseURL "<pages-url>/"`.

**См.** [`architecture/build-and-deploy.md`](../architecture/build-and-deploy.md).

## ADR-5. Никакого сборочного конвейера {#adr-no-pipeline}

**Решение.** Нет JS/CSS-фреймворка, нет npm/node, нет Hugo Pipes/бандлера. Только
чистый HTML/CSS/JS и Go-шаблоны. Минификация — флагом `hugo --minify` на CI.

**Почему.** Сайт — несколько статических страниц. Любой конвейер добавляет точки
отказа, CI-время, обновления зависимостей, уязвимости, когнитивную нагрузку — ради
нулевой пользы на этом объёме.

**Последствия.**
- Поиск — самописный клиентский фильтр по JSON, не библиотека.
- Иконки — инлайн-svg, не icon-библиотека.
- CSS — один файл с design tokens (CSS-переменные), не фреймворк.
- Новую зависимость тянуть **нельзя** без явной просьбы.

**См.** [`architecture/tech-stack.md`](../architecture/tech-stack.md).

## ADR-6. Явные ASCII-якоря заголовков {#adr-explicit-anchors}

**Решение.** Каждый заголовок поста — с явным ASCII `{#slug}`. Формат проверяет
render-hook (`errorf`), уникальность — `posts/single.html` (`errorf`).

**Почему.** Авто-id Goldmark из кириллицы percent-энкодится, `href` не совпадает с
`id`, скроллспай ToC (`navigation.js`, lookup `a[href="#id"]`) ломается. Явный
ASCII-якорь гарантирует совпадение.

**Последствия.**
- Забытый/не-ASCII якорь — build падает (это фича: ошибку видно сразу).
- Проверка уникальности вынесена из render-hook в `single.html`, т.к. `Page.Scratch`
  протекал между рендер-проходами дев-сервера.

**См.** [`components/posts.md`](../components/posts.md),
[`conventions/naming.md`](../conventions/naming.md#якоря-заголовков-).

## ADR-7. Серии — content section, а не taxonomy {#adr-series-section}

**Решение.** Серии реализованы как рукописный раздел `content/series/`, а не как
Hugo taxonomy (как теги).

**Почему.** Каждая серия — страница со своим заголовком, описанием и ручным порядком
постов (как `about.md`), а не авто-список. Taxonomy не даёт рукописного текста и
ручного порядка.

**Последствия.**
- Страницу серии создают вручную (`content/series/<name>.*.md`).
- Порядок — по `series.number` (ручной), не по дате.
- Нужны `url:` + `outputs: [html]` + гашение авто-`/series.html` через `_index`.

**См.** [`components/series.md`](../components/series.md),
[`patterns/content-section-pattern.md`](../patterns/content-section-pattern.md).

## ADR-8. Build-time валидация через `errorf`/`warnf` {#adr-build-validation}

**Решение.** Инварианты контента проверяются прямо в шаблонах: нарушение —
`errorf` (падает build) или `warnf` (предупреждение).

**Почему.** Ошибку контента лучше поймать на сборке, чем получить тихо сломанную
страницу. Дешевле любых внешних тестов, которых у контентного сайта нет.

**Последствия (текущие проверки).**
| Проверка                        | Где                                          | Уровень |
|---------------------------------|-----------------------------------------------|---------|
| Формат якоря заголовка          | `render-heading.html`                        | errorf  |
| Уникальность якоря в файле      | `posts/single.html` (через `assert-unique.html`) | errorf  |
| `series` без `name`/`number`    | `posts/single.html`                          | errorf  |
| Дубликат `series.number`        | `series/single.html` (через `assert-unique.html`) | errorf  |
| `series.name` без страницы серии| `posts/single.html`                          | warnf   |

Новые инварианты добавляйте тем же приёмом. Проверка «уникальность ключа в списке»
(якоря, номера серий) вынесена в общий partial `layouts/partials/assert-unique.html` —
он копит `seen`-слайс и падает `errorf` с готовым текстом ошибки, переданным
вызывающим шаблоном (сам partial не знает специфику сообщения, только механику
дубль-проверки). Новую проверку такого же вида (дубликат ключа) добавляйте через
него, а не копированием цикла `if in $seen / errorf / append`.

**См.** [`conventions/naming.md`](../conventions/naming.md).
